# 1. 배포 개요

Pullit 서비스의 전체 배포 아키텍처와 CI/CD 파이프라인의 흐름을 설명합니다. **이후의 문서들은 실제 배포 과정을 단계별로 상세히 다루므로, 기술적인 흐름에 관심 있으신 분들이 읽어보시면 좋습니다.**

## 1. 배포 철학: "Build on CI, Run on EC2"

우리 팀은 **"빌드는 CI 서버(Github Actions)에서, 실행은 운영 서버(EC2)에서"** 라는 명확한 원칙을 따릅니다. 이 '무빌드(No-Build) 배포' 방식은 서버의 부담을 최소화하고, 배포 속도를 극대화하며, 보안을 강화하는 현대적인 배포 전략입니다.

### 배포 흐름도

```mermaid
flowchart TD
    subgraph CI [CI 파이프라인 - GitHub Actions]
        A[1. 코드 빌드 & 테스트] --> B[2. Docker 이미지 생성];
    end

    subgraph Registry [Docker Hub]
        C[Docker 이미지 저장];
    end

    subgraph CD [CD 파이프라인 (GitHub Actions)]
        D[3. EC2에 접속] --> E[4. 최신 이미지 Pull];
    end
    
    subgraph EC2 [EC2 서버]
        F[5. 컨테이너 실행];
    end

    B --> C;
    C --> D;
    E --> F;
```

### 핵심 기대효과

-   **⚡️ 압도적인 배포 속도**: EC2 서버는 이미 빌드된 Docker 이미지를 내려받아 실행하기만 하면 되므로, 무거운 빌드 과정이 생략되어 배포가 수 분 내로 완료됩니다.
-   **💰 서버 자원 효율화**: 서버 내에서 직접 코드를 빌드하지 않으므로 CPU와 메모리 같은 값비싼 리소스를 빌드가 아닌, 오직 서비스 운영에만 집중시킬 수 있습니다.
-   **🛡️ 강화된 보안**: 운영 서버에 JDK, Gradle 등 빌드에 필요한 도구나 소스 코드를 둘 필요가 전혀 없습니다. 이는 공격자가 파고들 수 있는 잠재적 취약점을 원천적으로 차단하는 효과를 가집니다.
